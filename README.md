# Minecraft Server (Docker)

A self-built Docker image for a vanilla Minecraft Java Edition server, operated with
Docker Compose — fully configurable through environment variables, with persistent
world data in a named volume and automatic restart on crashes.

## Description

This repository contains everything needed to build and run a Minecraft server as a
Docker container. The image is **built from scratch** on top of a plain Java runtime
(`eclipse-temurin`) — no prefabricated Minecraft image is used:

- The official vanilla server JAR is downloaded from Mojang **at build time**
  (pinned via the download URL in the `Dockerfile`, so builds are reproducible).
- A small entrypoint script handles the Mojang EULA acceptance and starts the
  server with configurable JVM memory settings.
- All configuration lives **outside** the image: every setting — operational and
  gameplay — comes from a local `.env` file. At container start, the entrypoint
  renders `server.properties` from a template (`server.properties.example`) with
  `envsubst`, using the injected environment. The image itself contains no secrets
  and no environment-specific values.

At runtime, Docker Compose wires everything together: it builds the image, publishes
the server port, attaches a named volume for persistent world data, injects the
environment configuration and restarts the container automatically if the server
process crashes.

## Repository Contents

| File | Purpose |
| --- | --- |
| `Dockerfile` | Builds the server image: Java runtime base, installs `envsubst`, downloads the vanilla `server.jar` from Mojang, adds the entrypoint script and the config template, defines safe default environment values. |
| `entrypoint.sh` | Container startup script: writes `eula.txt`, renders `server.properties` from the template with `envsubst`, then starts the server with the configured memory settings. |
| `docker-compose.yaml` | Runtime configuration: builds the image, publishes the port, attaches the named data volume, injects `.env`, sets the restart policy. |
| `.env.example` | Template for the local `.env` file (all configuration). Copy it to `.env` and adjust the values — the real `.env` is excluded from git. |
| `server.properties.example` | Gameplay configuration template. Baked into the image and rendered to `/data/server.properties` at every container start; `${...}` placeholders are filled from the environment. |
| `.gitattributes` | Enforces LF line endings for shell scripts, so the entrypoint keeps working when the repository is checked out on Windows. |
| `.gitignore` | Keeps environment-specific files (`.env`) out of the repository. |
| `README.md` | This documentation. |

World data, logs and all files generated by the server live in the named Docker
volume `mc-data`, which is created automatically on first start.

## Quickstart

**Prerequisites:** Git, Docker Engine with the Compose plugin (Docker Desktop on
Windows/macOS, or Docker CE on Linux).

```sh
# 1. Get the code
git clone <repository-url>
cd <repository-folder>

# 2. Create your local configuration from the template
cp .env.example .env      # set EULA=true to accept the Mojang EULA

# 3. Build the image and start the server
docker compose up -d --build

# 4. Watch the startup (Ctrl-C detaches from the log, the server keeps running)
docker compose logs -f
```

Once the log shows `Done (…s)! For help, type "help"`, connect with your Minecraft
Java client to `<your-server-ip>:8888` (or `localhost:8888` when running locally).

> **Note:** The server only starts if you set `EULA=true` in your `.env`, which
> confirms that you accept the [Minecraft EULA](https://aka.ms/MinecraftEULA).

## Usage

### Environment Variables (.env)

Operational settings are injected at container start via the `.env` file
(`env_file:` in `docker-compose.yaml`). The image ships with safe defaults, so the
container starts even with an empty configuration — values in `.env` override them:

| Variable | Default | Description |
| --- | --- | --- |
| `EULA` | `false` | Must be set to `true` to accept the Mojang EULA — the server refuses to start otherwise. Deliberately **not** defaulted to `true`: accepting the EULA is an explicit operator decision. |
| `INIT_MEMORY` | `1G` | Initial JVM heap size (`java -Xms`). |
| `MAX_MEMORY` | `2G` | Maximum JVM heap size (`java -Xmx`). Keep this below the host's available RAM (rule of thumb: host RAM minus ~1 GB). |
| `HOST_PORT` | `8888` | Port published on the host. Inside the container the server always listens on 25565. |
| `TZ` | – | Timezone for log timestamps, e.g. `Europe/Berlin`. |
| `MOTD` | `A Minecraft Server` | Server name shown in the client's server browser. |
| `DIFFICULTY` | `easy` | Game difficulty: `peaceful`, `easy`, `normal` or `hard`. |
| `GAMEMODE` | `survival` | Default game mode: `survival`, `creative` or `adventure`. |
| `MAX_PLAYERS` | `20` | Maximum number of simultaneous players. |

### Game Settings (server.properties)

The gameplay configuration is **generated, not edited by hand**: at every container
start, the entrypoint renders `/data/server.properties` from the template
`server.properties.example` (baked into the image) using `envsubst`. The template
contains `${...}` placeholders for the configurable keys:

```properties
motd=${MOTD}
difficulty=${DIFFICULTY}
gamemode=${GAMEMODE}
max-players=${MAX_PLAYERS}
```

The values come from the container environment (see the table above), so the `.env`
file is the **single source of truth** for all configuration. Every placeholder has
a default baked into the image — this matters because `envsubst` replaces undefined
variables with an empty string, which would produce a broken config.

All other `server.properties` keys are fixed in the template. To change one of them
(or make it configurable), edit `server.properties.example` and rebuild the image.
Manual edits inside the running container are overwritten on the next start — by design.

### Applying Changes

Which command you need after a change depends on **where the changed value lives**:

| What changed | Command | Why |
| --- | --- | --- |
| `.env` or `docker-compose.yaml` | `docker compose up -d` | Runtime configuration is fixed when the container is created, so the container must be recreated. The entrypoint then re-renders `server.properties` with the new values. |
| `Dockerfile`, `entrypoint.sh` or `server.properties.example` | `docker compose up -d --build` | These are baked into the image, so the image must be rebuilt. |

To change the **Minecraft version**, replace the download URL in the `Dockerfile`
(`ADD https://… /opt/app/server.jar`) with the current server download link from
[minecraft.net](https://www.minecraft.net/en-us/download/server) and rebuild.

### Data & Persistence

The world lives in the named Docker volume `mc-data`, mounted at `/data` inside the
container. The server writes everything (world, logs, `eula.txt`, player data)
there. As a result:

- The world **survives** restarts, container re-creation and image rebuilds —
  `docker compose down` does *not* touch the volume.
- **`docker compose down -v` deletes the volume including the world** — only use
  it when you really want to start over.
- The data is managed by Docker; to look inside or copy files out, go through the
  container: `docker compose exec mc-server ls /data` or
  `docker cp mc-server:/data/logs/latest.log .`
- A simple backup while the server is stopped (Compose prefixes the volume with
  the project name — check `docker volume ls` for the exact name):
  `docker run --rm -v <project>_mc-data:/data -v "$PWD:/backup" ubuntu tar czf /backup/mc-data-backup.tgz -C /data .`

## Testing

Three checks to verify a deployment:

**1. Reachability** — from a different machine, with the
[mcstatus](https://github.com/py-mine/mcstatus) Python module:

```sh
pip install mcstatus
python -c "from mcstatus import JavaServer; s = JavaServer.lookup('<your-server-ip>:8888'); print(s.status().description)"
```

(or simply add the server in the Minecraft Java client and check the MOTD appears).

**2. Persistence** — destroy and recreate the container; the world must survive:

```sh
docker compose down && docker compose up -d
```

**3. Crash recovery** — kill the server process *inside* the container (a plain
`docker kill` counts as a manual stop and intentionally does **not** trigger the
restart policy):

```sh
docker exec mc-server kill 1
docker ps   # the container should be back up with a fresh uptime
```
